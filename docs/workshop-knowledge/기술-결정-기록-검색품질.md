# 기술 결정 기록 — 검색 품질 개선 전략

**결정일:** 2026-04-22  
**결정자:** 프로덕트앱개발팀  
**상태:** 구현 완료

---

## 문제

v1.0 Recall@5 40.2% — 사용자가 자연어로 질문할 때 CQL 단순 키워드 매칭의 한계 명확.

실패 케이스 유형:
- **SYNONYM_GAP:** "PR 리뷰 규칙" → "Pull Request 리뷰 가이드" (약어 미인식)
- **LLM_GENERATED:** "배포는 어떻게 해?" → "iOS 배포 프로세스" (LLM 생성 표현)
- **ABBREVIATION:** "아키 TF 자료" → "Architecture TF Weekly" (한국어 약어)
- **DATE_CONTEXT:** "4월 24일 미팅" → "Architecture TF Weekly @ 2026/04/24" (날짜 포맷 불일치)

## 검토한 옵션

### 옵션 A: 임베딩 기반 의미 검색 (RAG)
- 장점: 의미적 유사도 잡아냄
- 단점: ChromaDB 인프라 필요, 초기 인덱싱 비용, 응답 지연

### 옵션 B: Router LLM으로 쿼리 확장
- 장점: 인프라 추가 없음, CQL 기존 활용, 즉시 배포 가능
- 단점: LLM 호출 1회 추가 (비용/지연 소폭 증가)

### 옵션 C: 사전(dictionary) 기반 동의어 매핑
- 장점: 결정적(deterministic), 빠름
- 단점: 사전 유지보수 부담, 새로운 약어 대응 불가

## 결정: 옵션 B (Router LLM) 채택

즉시 배포 가능하고 인프라 의존 없음. RAG(옵션 A)는 나중에 fallback으로 추가.

## 구현 상세

### SYNONYMS 6가지 유형
```
1. 수식어 포함: 플랫폼·컨텍스트 붙인 원래 표현
2. 단축/핵심: 수식어 뺀 핵심 단어
3. 의미 동의어: 같은 개념의 다른 표현
4. 영문 변환: 한국어 → 영어 (문서 제목이 영문일 수 있음)
5. 약어 확장: PR → Pull Request, TF → 태스크포스
6. 날짜 포맷: 4월 24일 → 2026/04/24, 04/24, 4/24
```

### DATE_AFTER / DATE_BEFORE 분리 이유
날짜 의도가 두 종류임:
- **특정 날짜 문서:** "4월 24일 미팅" → SYNONYMS에 날짜 포맷 포함 (OR 제목 매칭)
- **최신 문서:** "최근 변경된" → DATE_AFTER CQL 필터 사용

이 둘을 하나로 처리하면 "4월 24일 미팅"이 DATE_AFTER로 분류되어 전혀 다른 결과 반환.

- MRR: **0.83**
- 평균 레이턴시: 3.2초
