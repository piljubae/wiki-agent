## 문제

코드 검색 쿼리 "클라이언트에서 API 호출 시 직접 세팅해서 보내는 custom request header" 같은 자연어 질문에서 **결과 0건** 반환. 3개의 연쇄 실패가 원인.

### 실패 1: ChromaDB `query_embeddings` 누락

`embeddingFn`이 null이거나 실패하면 `queryTexts`로 fallback했으나, 이 컬렉션은 서버 내장 임베딩 함수가 없어서 `queryTexts` 자체를 거부.

```
ERROR ChromaClient - ChromaDB query error: missing field `query_embeddings`
```

### 실패 2: GitHub Code Search URL 인코딩

한국어 자연어 문장이 URL 인코딩 없이 그대로 GitHub API에 전달. Ktor CIO가 non-ASCII 바이트를 처리 못해 HTTP 요청 자체가 실패.

```
WARN GitHubCodeClient - Failed to parse request body: request body length should be specified
```

### 실패 3: 자동 생성 동의어 품질

라우터가 SYNONYMS를 비워서 반환하면 `extractKeywordsAsSynonyms`가 bigram을 자동 생성. `클라이언트에서 *API`, `*API 호출` 같은 의미 없는 조합이 만들어짐.

## 결정 및 구현

### 1. ChromaDB queryTexts fallback 제거

**결정:** `queryTexts` fallback 경로 2개 모두 제거. embeddingFn 실패/null 시 `emptyList()` 반환, BM25+grep으로 우회.

**이유:** 이 프로젝트의 ChromaDB 컬렉션(`code_index`, `code_prs`)은 모두 외부 임베딩으로 생성. `queryTexts`가 정상 동작하는 컬렉션이 현재 0개.

**범위:** `CodeSearchTool.kt`, `PrHistoryTool.kt`

### 2. GitHub Code Search URL 인코딩

**결정:** `URLEncoder.encode()` 적용 + GitHub 검색 문법 특수문자(`*"()[]{}`) 제거. `repo:` qualifier는 인코딩 밖에서 직접 append.

**범위:** `GitHubCodeClient.kt` `searchCode()`

### 3. expandedQuery를 grep/GitHub 검색에도 활용

**결정:** `LlmExpandClient.expandQuery()` 결과(한+영 혼합)를 ChromaDB/BM25뿐 아니라 grep과 GitHub search에도 활용.

**변경 전:**
```
expandedQuery → ChromaDB, BM25만
query (원본)  → grep, GitHub (자연어 그대로)
```

**변경 후:**
```
expandedQuery → ChromaDB, BM25, grep, GitHub 전부
```

`extractEnglishKeywords()` 함수 추가 — expandedQuery에서 영문 키워드만 추출해 GitHub search에 전달.

### 4. 답변 프롬프트 toolName별 분기

**결정:** 코드 검색 결과인데 "회사 내부 위키 검색 봇" 프롬프트가 감싸는 문제 수정.

| toolName | 프롬프트 |
|----------|---------|
| codeSearch, codeStats, findCallers 등 | "소스코드 검색 봇" |
| prHistory | "PR 이력 검색 봇" |
| 그 외 | "회사 내부 위키 검색 봇" (기존) |

**범위:** `OrchestratorAgent.kt` 3단계 답변 생성

### 5. GitHub Code Search 청크 분할 fallback

**결정:** GitHub search는 공백 = 암묵적 AND. 8개 단어 AND면 거의 항상 0건.
AND 먼저 → 0건이면 3개씩 청크 분할 재시도. 키워드 순서 유지(expandQuery의 자연 순서 = 의미 클러스터).

**이유:** Confluence keyword fallback의 AND→OR 재시도 패턴과 동일한 접근. (참고: `project_keyword_search_and_or.md`)

**범위:** `CodeSearchTool.kt` `searchGitHubWithChunkedFallback()`

### 6. grep 패턴 LLM 전용 호출 분리

**결정:** `buildGrepPatterns(expandedQuery)` 휴리스틱이 `Custom`, `User`, `Type` 같은 너무 일반적인 PascalCase 단어를 추출해 엉뚱한 파일 매칭. LLM에게 쿼리 의도에 맞는 코드 식별자를 직접 요청.

**변경 전:** `buildGrepPatterns()` — PascalCase/camelCase 정규식 추출 (쿼리 의도 무시)
**변경 후:** `LlmExpandClient.extractGrepPatterns()` — LLM이 쿼리 의도에 맞는 식별자만 반환

`expandQuery`(임베딩용)와 병렬 실행 — 추가 지연 없음. LLM 실패 시 기존 `buildGrepPatterns()` fallback.

**기대 결과:**
```
변경 전: grep patterns=[API, HTTP, Custom, User, Agent, Type] → LikeCountButton.kt (무관)
변경 후: grep patterns=[addHeader, setHeader, Interceptor, OkHttpClient] → 실제 헤더 설정 코드
```

### 7. config 설정 복원

**문제:** config.yml에서 `codeSearch.embeddingMode: GOOGLE_EMBEDDING`과 `localRepoPath` 설정이 누락되어 3대 백엔드(벡터/BM25/grep)가 전부 비활성화 상태였음.

**조치:**
- `embeddingMode: GOOGLE_EMBEDDING` 추가
- `localRepoPath: /Users/pilju.bae/.wiki/index/kurly-android` 추가
- `config.yml.template` 레포에 추가 — 설정 누락 재발 방지

## Confluence keyword fallback AND/OR 결정 (2026-05-28)

### 문제

Confluence keyword 검색에서 AND/OR 중 하나를 고정하면 근본적으로 부족.

- **OR 단독 (과거):** "코드리뷰 스킬" 검색 시 상품 리뷰 관련 문서가 상위에 오는 false positive
- **AND 단독 (`isConversationalQuery` 도입 후):** "온보딩 프로세스" 검색 시 제목이 "신규 입사자 온보딩"이고 본문에 "프로세스"가 없어 0건
- **`isConversationalQuery` 휴리스틱:** 조사/어미 여부 + 키워드 비율로 판단했으나 명사 2개 조합도 실패

### 결정

AND로 먼저 시도, 결과 0건이면 OR로 재시도.

### 범위

`ConfluenceSearchAgent.kt` keyword fallback 로직
