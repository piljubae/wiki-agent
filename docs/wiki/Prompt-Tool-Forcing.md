# 프롬프트 설계 원칙 3 — Tool 호출 강제 vs 직접 답변

## 두 가지 전략

### 전략 A — Tool 호출 강제

```
"사용자의 질문에 답하기 위해 반드시 제공된 Tool을 사용해 검색하세요.
검색 없이 직접 답변하지 마세요."
```

### 전략 B — 직접 답변 허용

```
"알고 있는 경우 바로 답하고, 모르는 경우 Tool을 사용해 검색하세요."
```

## wiki-agent는 왜 전략 A를 쓰나

wiki-agent의 목적은 **회사 내부 문서**를 찾아주는 것입니다.

| 상황 | 전략 A 결과 | 전략 B 결과 |
|------|-----------|-----------|
| "배포 프로세스 알려줘" | 실제 사내 위키 검색 | LLM이 일반적인 배포 절차 직접 답변 |
| "온보딩 체크리스트" | 사내 온보딩 문서 검색 | LLM이 일반 온보딩 내용 직접 답변 |

전략 B를 쓰면 **우리 회사 문서가 아닌 LLM 학습 데이터** 기반 답변이 나옵니다.

## 전략 선택 기준

| 상황 | 추천 전략 |
|------|---------|
| 최신 내부 정보가 중요 (사내 위키, 운영 문서) | **A — Tool 강제** |
| 일반 지식도 유효 (FAQ, 기술 개념 설명) | **B — 직접 허용** |
| 정확도가 비용보다 중요 | **A — Tool 강제** |
| 응답 속도가 중요 | **B — 직접 허용** |

## 프롬프트 강도 조절

**강함 (wiki-agent 현재):**
```
"검색 없이 직접 답변하지 마세요."
```

**중간:**
```
"가능하면 Tool을 사용해 검색 후 답변하세요."
```

**약함:**
```
"필요한 경우 Tool을 사용할 수 있습니다."
```

강도가 약할수록 LLM이 Tool 없이 직접 답하는 빈도가 높아집니다.
