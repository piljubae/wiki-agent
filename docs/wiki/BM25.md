# BM25

"이 단어가 이 문서에서 얼마나 중요한가"를 점수로 매기는 키워드 검색 알고리즘.
1994년에 나왔고 지금도 Elasticsearch, Google 검색 등의 핵심 랭킹 알고리즘으로 쓰입니다.

---

## 핵심 아이디어 2가지

### 1. 단어 빈도 (TF) — 많이 나올수록 높은 점수, 단 수확 체감

```
문서 A: "panelCode" 1번  →  점수 1.0
문서 B: "panelCode" 5번  →  점수 2.3   (5배가 아님)
문서 C: "panelCode" 20번 →  점수 2.8   (20배가 아님)
```

처음 등장은 중요하지만, 반복될수록 추가 가중치는 급격히 줄어듭니다.
"이 단어를 20번 언급한 문서"가 "1번 언급한 문서"보다 20배 더 관련 있지는 않기 때문입니다.

### 2. 희귀도 (IDF) — 드문 단어일수록 높은 점수

```
"panelCode" → 전체 32,000 청크 중 50개에만 등장  →  희귀  →  가중치 높음
"fun"       → 거의 모든 청크에 등장              →  흔함  →  가중치 낮음
"val"       → 모든 청크에 등장                   →  매우 흔함  →  가중치 거의 0
```

전체 문서에서 드물게 등장하는 단어일수록 특정 문서를 구별하는 힘이 셉니다.

| | 벡터 검색 | BM25 |
|-|---------|------|
| "배너 클릭 이벤트 처리" | ✅ 의미로 찾음 | 단어 분산돼 약함 |

| 오타, 신조어 | 어느 정도 허용 | ❌ 정확히 일치해야 함 |

## wiki-agent 구현 — SQLite FTS5

BM25는 역인덱스가 필요합니다. Elasticsearch 같은 별도 서버 없이
SQLite 내장 FTS5(Full-Text Search 5)를 쓰면 파일 하나로 끝납니다.

```kotlin
// BM25Index.kt — 테이블 생성
stmt.execute("""
    CREATE VIRTUAL TABLE IF NOT EXISTS code_chunks USING fts5(
        id UNINDEXED,
        content,
        tokenize = 'unicode61'
    )
""")

// 검색: bm25() 점수 기준 정렬 (FTS5 내장 함수, 음수 반환 → ASC 정렬)
"SELECT id FROM code_chunks WHERE code_chunks MATCH ? ORDER BY bm25(code_chunks) LIMIT ?"

// upsert: FTS5는 UPDATE 미지원 → DELETE + INSERT
fun upsert(id: String, content: String) {
    conn.prepareStatement("DELETE FROM code_chunks WHERE id = ?").use { ... }
    conn.prepareStatement("INSERT INTO code_chunks(id, content) VALUES (?, ?)").use { ... }
}
```

## unicode61 토크나이저와 한국어 한계

FTS5가 문서를 단어로 쪼갤 때 사용하는 규칙입니다.

```
"panelCode onBannerClick KMA-7275"
  → 분리 결과: ["panelCode", "onBannerClick", "KMA", "7275"]  ✅

"배너 클릭 이벤트 처리"
  → 분리 결과: ["배너", "클릭", "이벤트", "처리"]  ✅ (공백 기준)

"배너클릭이벤트"  (공백 없음)
  → 분리 결과: ["배너클릭이벤트"]  ❌ (형태소 분리 안 됨)
```

unicode61은 공백/구두점 기준으로만 쪼갭니다.
한국어는 형태소 분석기(Nori, Mecab 등)가 있어야 "배너클릭이벤트" → ["배너", "클릭", "이벤트"]로 분리됩니다.

**실용적 결론**: wiki-agent에서 한국어 쿼리는 BM25보다 벡터 검색이 더 효과적.
영어 코드 키워드(`panelCode`, `KMA-7275`, `onBannerClick`)에서 BM25 효과가 발휘됩니다.
