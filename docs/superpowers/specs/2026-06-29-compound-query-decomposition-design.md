# 복합 질문 분해 라우팅 설계 (Compound Query Decomposition Routing)

- 작성일: 2026-06-29
- 상태: 설계 확정
- 범위: `OrchestratorAgent` 라우팅 레이어 (전 tool 공통)

## 1. 배경 / 문제

현재 라우팅은 **질문 1개 → 라우터 LLM 1회 → 13개 tool 중 1개 선택**이다. 복합 대응은
하드코딩된 조합 2개(`confluenceSearch+codeSearch`, `prHistory+codeSearch`)뿐이며, 그마저도
두 tool에 **같은 query**를 넣는다.

그 결과 "결제 흐름 코드랑 관련 위키 문서, 최근 PR도 알려줘" 같은 복합 질문을 부위별로
서로 다른 에이전트에 서로 다른 쿼리로 보내지 못한다.

사용자 목적: **복합 질문이 들어오면 의도를 먼저 분석·분해하고, 각 조각을 알맞은 에이전트로
보내 각자 결과를 리턴받아, 섹션별로 합친 통합 답변을 만든다 — Confluence뿐 아니라 전 tool에 걸쳐.**

관련 1차 진단(의도 파악 약점): ① 질문→키워드 변환 부정확, ② 키워드는 맞아도 CQL이 못 찾음(의미
격차), ③ 복합·다단계 질문 미분해. 본 설계는 **③(과 ①의 일부)** 를 해결한다. ②(의미/벡터 검색
도입)는 별도 후속 프로젝트로 분리한다.

## 2. 접근 결정 (경로 Y)

라우터를 통째로 "플래너"로 교체(경로 X)하는 대신, **분해기를 한 겹 앞에 두고 기존 라우터를
sub-question마다 재사용**한다(경로 Y).

| 기준 | 경로 X (JSON 플래너) | 경로 Y (분해기 + 라우터 재사용) ★채택 |
|------|--------------------|----------------------------------|
| 추가 LLM 호출 | 없음 | 복합질문일 때 분해 1 + 라우팅 N (라우팅은 병렬) |
| 영향 범위 | 13개 tool 전부 | 신규 분해기 1개로 격리 |
| 실패 시 | 모든 검색 위험 | 폴백으로 기존 동작 100% 보존 |
| 작업량 | 큼(전체 회귀) | 작음 |

채택 이유: 검증된 라우터를 재작성하지 않고 재사용하며, 신규 위험이 분해기 하나로 격리된다.
효과 확인 후 필요하면 경로 X로 합칠 수 있다.

## 3. 전체 흐름

```
질문
 │
 ├─ 프리라우트(기존): 인사·코칭·온보딩·none → 분해 없이 기존대로 처리
 │
 ▼ (검색성 질문만)
[QueryDecomposer] 질문 → sub-question 1~N개
 │   단순질문이면 1개(=원본) → 사실상 기존 동작
 ▼
[기존 라우터 재사용] 각 sub-question → tool+query+synonyms 결정 → 검색   ← 병렬
 │   (라우터가 조합 tool을 고르면 한 조각이 여러 에이전트로 자동 fan-out)
 ▼
[집계] sub-question별 결과 라벨링 + pageId/url 중복제거
 ▼
[요약 LLM] 섹션별로 정리한 통합 답변
```

## 4. 컴포넌트

### 4.1 (신규) QueryDecomposer
- 책임: 질문(+최근 대화맥락)을 1~N개의 독립 sub-question으로 분해한다.
- 입력: `question: String`, `context: String`(최근 대화 일부).
- 출력: `List<String>` (sub-question 문자열). 단순질문은 원본 1개, 복합질문은 2~3개.
- 구현: LLM 1회 호출. 작은 `routerModel`을 재사용해 비용·지연을 낮춘다.
- 출력 포맷: JSON 배열 권장. 파싱은 `runCatching`으로 감싸고 실패 시 `[원본질문]` 폴백.
- 맥락 반영: "그거 코드도" 같은 후속 복합질문의 지시어 해소(보너스, 1-hop 한정).
- 스킵 휴리스틱: 매우 짧은 질문·접속 표현 부재 시 분해 호출을 생략하고 `[원본]` 사용 가능.
  (또는 분해기가 1개를 반환하면 추가 비용 없이 기존 경로로 수렴.)

### 4.2 (리팩터) 라우팅+검색 추출
- 현 `OrchestratorAgent.answer()`는 라우팅 → 검색 → 요약이 한 덩어리다.
- "질문 1개 → 검색결과 문자열" 부분(현 `OrchestratorAgent.kt:401~454` 영역)을
  `routeAndRetrieve(subQuestion, context): StepResult` 로 분리한다.
- 요약 단계(현 `OrchestratorAgent.kt:457~518`)는 그대로 두고, 단일/복합 모두 마지막에 한 번만 탄다.
- N개의 `routeAndRetrieve`는 코루틴으로 병렬 실행한다.

### 4.3 (변경) 집계 + 요약
- N개 `StepResult`를 sub-question 라벨과 함께 요약 프롬프트에 넣는다.
- 요약 프롬프트에 "각 항목을 섹션으로 나눠 답하라"는 지시를 추가해 통합 섹션형 답변을 생성한다.
- 집계 시 pageId/url 기준 중복 결과를 제거해 같은 문서가 여러 섹션에 반복되지 않게 한다.

## 5. 데이터 형태

```kotlin
data class StepResult(
    val subQuestion: String,
    val toolName: String,
    val searchResult: String?,   // null = 못 찾음
)
```

## 6. 매핑 규칙 (1 sub-question → 필요시 여러 에이전트)

- 각 sub-question은 **기존 라우터**가 tool을 결정한다.
- 라우터가 조합 tool(`confluenceSearch+codeSearch`, `prHistory+codeSearch`)을 고르면 그 조각은
  자동으로 여러 에이전트로 fan-out된다. 별도 fan-out 기계장치는 두지 않는다.
- sub-question이 검색성이 아닌 특수 tool(`none`/`onboarding`/`progressAdvisor`)로 라우팅되면
  해당 조각은 드롭한다(복합 검색 맥락에서 특수 플로우 혼입 방지). 특수 플로우는 §7 참조.

## 7. 실패·지연·격리 처리

- 분해기 JSON 깨짐/빈 결과 → `[원본질문]` 폴백 = **기존 동작 100% 보존**.
- 전체 질문이 프리라우트에서 특수 tool(인사·코칭·온보딩·none)로 잡히면 **분해 자체를 건너뛴다**.
- 특정 sub-question 검색 실패 → 그 섹션만 "못 찾음", 나머지 섹션은 정상 진행.
- 단순질문(분해 결과 1개) → 추가 라우팅 호출 없이 기존 단일 경로와 동일.
- N개 라우팅 병렬 → 지연 = (분해 1회) + (가장 느린 라우팅 1개).

## 8. 테스트 (testing-strategy L1 중심)

- L1(Unit, mock LLM):
  - 분해기 출력 파싱: 복합→N개 / 단순→1개 / JSON 깨짐→`[원본]` 폴백.
  - 집계 중복제거: 동일 pageId/url 결과 dedup.
  - 섹션 조립: N개 StepResult → 섹션 라벨 포함 요약 프롬프트 구성.
  - 특수 tool 드롭 규칙.
- 회귀:
  - 분해기가 1개를 반환할 때 단일질문 경로가 기존과 동일한 결과/호출 수인지.
  - 프리라우트 특수 tool 경로가 변화 없는지.

## 9. 범위 밖 (YAGNI)

- 라우터 출력 포맷 JSON 전면 교체(경로 X).
- 의미(벡터) 검색 도입(접근법 B) — 별도 후속.
- sub-question 간 의존(A 결과로 B 질문 생성) 등 다단계 추론 — 분해는 1-hop 병렬만.
